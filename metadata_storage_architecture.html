Metadata Storage Strategy:
Relational Database vs. Semantic Vector Store

Definitive architectural division governing physical metadata storage, semantic discovery schemas, and dispatcher routing pipelines.

Author: Anvizent AI Architecture Team

01 The Governing Rule

When building a semantic natural-language assistant to query multi-source databases, a common architectural pitfall is storing all metadata in a vector store and relying on LLM embeddings to construct queries and joins. In enterprise production, this causes query failures due to halluncinations, lack of strict join path mapping, and missing data types.

✅ DO: Store structural metadata in a relational database and perform SQL lookups.
Use the relational database to fetch exact table definitions, schema column names, primary/foreign key join relationships, business metrics formulas, and data quality check constraints. This guarantees 100% deterministic precision.
❌ DO NOT: Rely on vector searches to retrieve schemas, column mappings, or join paths.
Vector store lookups are non-deterministic and suffer from loss of structured context. An LLM querying a vector store for schemas cannot reliably reconstruct complex join syntax or data type alignment.

The Strategy: Use the Vector Store exclusively for initial semantic discovery (i.e., mapping the user's plain-English terms to canonical business entities or KPIs). Once the candidate entities are discovered, perform a SQL metadata lookup against the relational catalog to retrieve the exact physical schemas, connections, and join syntax required for the query.

02 Relational Catalog (metadata_catalog_db)

The metadata_catalog_db is the source of truth for the system's structural integrity. It maintains strict relations, datatypes, and constraints.

🔌 1. Ingestion Layer

Maintains physical credentials, host endpoints, and table schemas of source databases.

  • source_connections: Extracted system paths, ports, types.
  • physical_metadata: Source-faithful raw column and data type schemas.

🧠 2. Semantic Catalog Layer

Maps source columns to standardized vocabulary and defines metrics, business rules, and constraints.

  • standardized_attributes: Column-to-attribute mapping.
  • semantic_entities: Standardized objects (e.g. Employee).
  • semantic_attributes: Canonical business fields.
  • semantic_metrics & mappings: KPI aggregate formulas.
  • semantic_relationships: Primary/Foreign key paths.
  • semantic_business_rules: Computations (e.g. Net Rev).
  • semantic_glossary & synonyms: Business definitions.

🏗️ 3. Warehouse, Mart, & Lineage Layer

Defines the target analytical schemas and transformation jobs representing how data is processed.

  • dwh_dimensions_facts: Star Schema dimensions (`dim_employee`) and facts (`fct_absences`) schemas.
  • dm_data_marts: Structured views / aggregates queried by business users.
  • etl_lineage: Direct source-to-target pipeline job dependency maps.

03 Database DDL Representation

The following DDL schemas represent the structured catalog tables in metadata_catalog_db:

-- Core Ingestion tables
CREATE TABLE source_connections (
    system_id VARCHAR(50) PRIMARY KEY,
    system_name VARCHAR(100) NOT NULL,
    system_type VARCHAR(50),
    host_endpoint VARCHAR(255),
    source_schema VARCHAR(100)
);

-- Standardized attribute crosswalk
CREATE TABLE standardized_attributes (
    attribute_id INTEGER PRIMARY KEY,
    business_attribute_name VARCHAR(100) NOT NULL,
    source_system VARCHAR(50) NOT NULL,
    physical_table VARCHAR(100) NOT NULL,
    physical_column VARCHAR(100) NOT NULL,
    datatype VARCHAR(50)
);

-- Join Paths definition
CREATE TABLE semantic_relationships (
    relationship_id INTEGER PRIMARY KEY,
    parent_entity VARCHAR(100) NOT NULL,
    child_entity VARCHAR(100) NOT NULL,
    parent_join_column VARCHAR(100) NOT NULL,
    child_join_column VARCHAR(100) NOT NULL,
    relationship_type VARCHAR(50) -- 'One-To-Many' / 'Many-To-One'
);

04 Semantic Vector Store (semantic_vector_store)

The semantic_vector_store is used only for **semantic matching and discovery**. It stores textual embeddings of concepts to map user phrasing to canonical metadata keys.

How it Works: When a user asks: "What is our worker count?", the vector store translates worker count to the canonical metric Employee Count and the entity Employee. It does not provide schemas; it simply returns the exact IDs/keys needed to query the relational database.

Vector Collections structure:

Collection NameMetadata Payload fieldsPurpose
Glossary { "term": "Employee", "domain": "HR" } Resolves user business jargon to standardized definitions.
Entities { "entity_id": 1001, "entity_name": "Employee Absence" } Maps phrasing like "leaves" or "vacation" to canonical entities.
Metrics { "metric_name": "Revenue", "aggregation": "SUM" } Resolves metric inquiries to canonical KPI metrics.
Business_Rules { "rule_id": "BR001", "entity": "Employee Absence" } Identifies when a question references calculated business logic.
Synonyms { "canonical_term": "Revenue", "synonym": "turnover" } Expands plain-text search terms using custom enterprise thesauruses.
