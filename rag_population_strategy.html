How knowledge gets into the RAG

Three input sources feed the knowledge store: standardized vendor metadata, industry-specific seed metadata, and the end-user Q&A that the Data Engineering Bot runs to build pipeline JSON. Two are load-ahead grounding; one is earned over time and compounds the other two.

The mental model

Four sources, four kinds of truth

Each input source carries a distinct kind of truth into the RAG. Keeping them distinct is what lets the bot ask business-language questions and still ground every suggestion.

SOURCE A

Structural truth

What exists. The published object/field/endpoint model of each software — stable, standardized, identical across every customer of that vendor.

SOURCE B

Semantic truth

What it means here. Industry-specific metric definitions, formulas, business rules, synonyms — the vertical's shared vocabulary.

SOURCE D

Self truth

Who this tenant is. The business's own context — offerings, economics, and how its data flows. Captured via business onboarding (Type 7); the day-one tenant grounding.

SOURCE C

Experiential truth

What actually works. The patterns, customizations, and outcomes earned from real end-user Q&A → deploy loops.

A & B make the bot competent. D grounds it in this tenant. C makes it expert. A, B and D are loaded ahead as grounding; C accumulates once the bot is live and flows back to enrich them. D is a living record, maintained across the customer lifecycle.

A · Vendor standardized metadata Gated · global

The published, version-stable structural docs for each software: data dictionaries, OpenAPI / API specs, ERDs, object/field references (Salesforce object reference, SAP table/field docs, Epicor data dictionary, and so on).

Acquire
Harvest each vendor's published structural docs per {system, version}. Standardized and public, so it loads once, globally — Salesforce's object model is identical for every Salesforce customer.
Transform
Parse → section-aware chunk (one object/table or one endpoint per chunk family) → embed → tag system, version, object_type, field.
Gate
Curated. Versioned; on a vendor version bump the prior version is superseded (retained while any pipeline is pinned to it, then hard-deleted).
Land
Type 1a → kb:sor-docs:{system}:{version}
Role
Gives the bot candidate tables/fields and their meanings so it can propose joins and lookups without exposing table names to the user.
Public docs are indexed; the live tenant instance schema is not. The connected instance schema (Type 1b) is introspected at query time and cached — never embedded — because it drifts per tenant and would rot the index. Stable docs = indexed reference; live schema = ephemeral cache.

B · Industry seed metadata Gated · global

Curated, vertical-specific generic knowledge: metric / KPI definitions, formulas, business rules, synonyms. (No canonical datamarts/patterns — those are earned via usage, not seeded.)

Acquire
Team-authored per vertical, seeded from industry docs — definitions, formulas, synonyms, business rules.
Transform
Author → review/approve → version → embed with definition, formula, industry tag, synonyms.
Gate
Human-curated, versioned. Corrected definitions supersede; prior kept read-only for lineage.
Land
Type 3 → kb:semantics:{industry}:{version} (generic semantics only).
Role
Lets the bot's BA-style questions reference business concepts ("net revenue", "absence rate") instead of columns. Canonicals emerge later from tenant usage (C11) — none on day one.

D · Business onboarding Gated · tenant · living

The tenant's own business context — what it is, what it sells, how it makes money, and how its data flows. The day-one grounding a new tenant supplies (since canonicals are emergent and Source B is generic). See the Business Onboarding spec.

Acquire
Business onboarding Q&A (business user); system detail added progressively as systems are brought in.
Transform
Capture identity / offerings / differentiation / economics; derive the data-flow stages and confirm them with the user (never hard-coded).
Gate
Human-supplied; a living record — lifecycle updates supersede with lineage retained.
Land
Type 7 → tenant DB (business_context / value_chain_stage / system_inventory) + discovery vector.
Role
Consulted first by the agent so it can suggest sources, stages and metrics — cutting the questions on every new pipeline build.

C · End-user Q&A (the Data Engineering Bot) Hot · tenant-scoped

The bot runs its fixed BA-style method and emits datamart / pipeline JSON. The method is an input, not redefined here — the strategy harvests what it produces. The agent is tenant-specific (C10). Each artifact lands in a different knowledge type.

Datamart JSON
→ Type 6 resolved semantics. The working datamart + DWs + columns, on confirmation + validation + usage (not deploy), recorded scope-tagged (company…user) and composable (sub-datamarts) in the tenant datamart catalog.
Pattern
→ Type 2. The generalized signature (scrub tenant values → generalize) → kb:patterns:global; becomes canonical only via repeated usage, never seeded.
Unknowns
If a source (incl. webhooks / FTP files) isn't in RAG, the agent asks the user to supply it or a doc link and loads it into Type 1a (C7).
Metrics
→ Type 3 suggestions. A new definition queues for batch review; if incongruent with an existing one, the agent asks why and records the reason in RAG (C8). Never auto-written.
Overrides
→ Type 4 customization. Diff vs the generated baseline → the tenant's own database (physically isolated).
Run telemetry
→ Type 5 insight. Rules gate → LLM phrases survivors; selection/success counts drive reinforcement and decay.
The engine

The bot is both consumer and producer

At build-time the bot consumes the RAG to ground its questions; at deploy-time it produces knowledge that flows back in. That two-way flow is the "populate and use" loop.

Build-time — the bot consumes

User Q&A→ retrieve A structure+ Type 1b live schema+ retrieve B semantics+ Type 2 prior patterns+ Type 4 tenant customizations→ grounded BA suggestion

Deploy-time — the bot produces

Pipeline JSON→ deploy + reconcile→ scrub + generalize→ Type 2 pattern · Type 3 suggestion · Type 4 customization · Type 5 telemetry

How they combine

One retrieval, three sources

At build-time

A single retrieval pulls across all three:

  • A offers candidate tables / fields
  • B supplies metric / KPI definitions and business rules
  • C supplies reusable datamarts (Type 6), proven patterns + this tenant's customizations

That combination is how the BA-framed questions get grounded, table-name-free suggestions.

On confirmed use, C flows back

  • Resolved datamarts grow Type 6 (and earn canonical status over time)
  • Generalized patterns grow Type 2
  • Metric definitions enrich Source B; run telemetry grows Type 5
A and B are loaded ahead; C compounds them on confirmation + validation + usage (not deploy). The store gets denser and more tenant-aware over time without anyone re-authoring the grounding.
Order of operations

Sequencing that falls out of the model

1 · Load A structural→ 2 · Load B generic semantics→ 3 · Turn on the bot — C compounds→ 4 · Canonicals emerge from usage
  • Load A (structural) and B (generic semantics) first — grounding the bot needs before its first question.
  • No canonicals on day one (C11). The cold start is accepted: the bot asks more early and learns. Canonical datamarts/patterns are earned as they recur across tenant usage.
  • Turn on the bot — Source C compounds Types 2 / 3 / 4 / 5 / 6 on confirmed-and-used builds.
Non-negotiable

Guardrails on population

Suggest, never auto-canonize

Q&A-surfaced definitions go to the batch-proposed-diff queue, never straight into the canonical glossary — or the bot manufactures consensus from a handful of tenants.

Scrub before global

Tenant-specific values are redacted and generalized before any pattern or insight is promoted to a global namespace.

Never index the live schema

The connected instance schema is cached at query time, never embedded — indexing drift is the quiet way the store rots.
