The gap it fills

Day-one tenant grounding

Per the reconciliation, there are no canonical patterns on day one and the industry seed is generic — so a brand-new tenant has no tenant-specific grounding. Business onboarding is how the customer supplies it, once, and keeps it current.

The payoff is question reduction. The data-flow map is the spine: a question about "win rate" → context says closing runs in CRM X → the agent suggests the source and entities instead of asking cold. Each onboarded fact removes a class of repeated questions on every future build.
Two core principles

What's fixed, and what isn't

1 · The data flow is canonical; the stages are not

Capturing the data flow is required for every business. The steps vary by business type, so they are not hard-coded — the agent derives them with basic questions and confirms with the user (suggest & confirm).

A common lifecycle is offered as a template, then adapted per business. The confirmed, business-specific stage list is what gets stored — never the template by assumption.

2 · Business context is a living record

Onboarding is the initial seed, not a freeze. Context is updatable throughout the customer lifecycle — new product lines, revenue streams, systems, changed processes.

Updates supersede prior versions (lineage retained), triggered by explicit re-onboarding or surfaced during a build ("you told us X served billing — has that changed?").

How it's captured

Two onboarding tracks

A · Business onboarding once, then maintained

Answered by the business user (owner of requirements).

🏷️ Identity

What the business is · website · industry / vertical.

📦 Offerings

Products / services / solutions delivered to customers.

⭐ Differentiation

Unique advantages; why customers choose them.

💰 Economics

How they make money (revenue model) and where margin comes from — margin drivers + the cost / COGS side.

🔀 Data flow

The derived-and-confirmed stage list; per stage: systems, key entities, handoffs.

B · System onboarding progressive

Answered by the engineer / analyst, captured the moment a system is added (progressive opt-in — not demanded upfront).

  • Which value-chain stage(s) it serves and its role
  • Key objects / identifiers and how records flow in & out
  • Connection method — API · webhook · FTP file · DB
  • If structure isn't known, the agent asks for it or a doc link and loads it into RAG (C7)
"As they want to get more systems involved, we onboard each system with a bit more detail" — the inventory deepens over time without a big upfront ask.
How context gets in

Intake — Q&A and document / link upload

Customers already have decks, brochures, contracts, and a website that describe the business. So context enters two ways — both feed the same suggest-&-confirm loop.

Supported formats

📄 PDF

Decks, brochures, contracts, annual reports — text parse, OCR for scans.

📝 Word

.docx overviews, SOPs, process docs — text + structure parse.

🖼️ Images

Logos, screenshots, org charts, scanned one-pagers — OCR + vision.

🔗 Website links

Homepage, About, Products, Pricing — fetch + readability extraction.

Extensible — spreadsheets, slides, plain text: any format → extract → propose → confirm.

The ingestion flow

Upload / link→ Extract (parse · OCR · vision · fetch)→ Propose candidates→ User confirms→ Type 7
Never auto-written. Extracted business facts are proposals; the business user confirms — same guardrail as everywhere (suggest & confirm).
Provenance. Every artifact is recorded (format, file / URL, status) and linked to the facts it produced — each confirmed field traces back to its source.
Living. Re-uploading an updated deck or re-fetching the site supersedes the prior extraction (lineage retained) — onboarding stays current.
Privacy. The user uploads / links; the platform never reaches into systems itself. Document content is data to extract, not instructions to act on.
The gate

Extraction quality validation = the user confirming

Quality scoring doesn't decide whether a field is accepted — it decides how the field is shown. Every field ends at the same gate: Confirm (accept) or Correct (edit). Nothing enters Type 7 without it.

What we score

  • Confidence — the model's extraction confidence (0–1)
  • Grounding — quoted from a span/page/URL vs inferred
  • Corroboration — do multiple sources agree, or conflict?
  • Plausibility — format & coherence checks

Disposition — all roads lead to the user

  • Confirm (high + grounded) → pre-filled, one-tap confirm
  • Review (medium / inferred) → draft, flagged to verify/edit
  • Ask (low / missing) → falls back to a question — don't guess
  • Conflict (sources disagree) → show both, user picks/corrects

Try the review queue

Four extracted fields, four dispositions. Confirm or correct each — only confirmed fields land in Type 7.

The gate is fixed; the thresholds are not. Confidence cutoffs for Confirm / Review / Ask are tunable config (the parameter registry, with the Locked / Learning toggle) — corrections become tuning signal — but no threshold ever removes the confirmation step.

The point

How it cuts the question count

During the agent build process, before asking, the agent reads Type 7:

  • Source applications (build step 1) — the system inventory already names the systems and the stages they serve → suggest, don't ask.
  • Which stage a metric belongs to — the data-flow map routes "revenue" → billing, "pipeline" → lead/closing → suggests the right system/entities.
  • Metric framing — the economics (revenue model, margin drivers) seed candidate metrics and their business meaning.
  • New systems — system onboarding pre-answers the structural questions for that system on every future build.
Each onboarded fact removes a class of repeated questions; the living updates keep it from going stale.
