New concepts

Type 6 — Resolved contextual semantics

The DW + datamart catalog the agent operates on. A datamart's working JSON + its DWs + the columns it uses are semantic knowledge. Vector chunks + a relational catalog, every record scope-tagged and composable via composes_of.

Scope taxonomy

company → department → purpose → role → user. Elicited by the agent and attached to every Type 6 record and to a scheduled purpose's confirmation.

Role ownership

Business = owner of requirements · Analyst = owner of definitions · Engineer = owner of deployments. Drives role-adaptive questions and who confirms what.

Reuse is compositional. A requirement decomposes into sub-datamarts; each is a reusable building block — composed into the current mart and reusable for other questions. Reuse = decompose → match the catalog → add only net-new DW tables/columns → compose.

The record

Decision detail

C1 Datamart JSON is semantic knowledge

The pipeline/datamart working JSON + DWs + columns used is stored as Type 6 resolved contextual semantics — the resolved context of the question, shared across the company — not merely a generalized Type 2 pattern.

C2 DW/DM knowledge is scoped; the agent elicits scope

Type 6 records carry a scope (company/department/purpose/role/user). The agent asks which scope applies and tags the record. This is the home for the DW + datamart catalog.

C3 Reuse is compositional

A requirement is broken into sub-datamarts; each is reusable to assemble the current mart or to answer other questions. The catalog stores composable units linked by composes_of; reuse adds only net-new DW tables/columns.

C4 Reconciliation lives in build + JSON validation

Flag unmatched dimension values as "Unknown" (never drop), row-level uniqueness before any join, date/time-range-only extraction, never pre-join, extract datasets individually — explicit build steps and enforced at JSON validation.

C5 Capture is usage-based, not deploy-based

"Deployment success means nothing." Confidence is driven by user confirmation + data validation + usage. Onetime → confirm narrowly, low reliance. Scheduled → a purpose is served (rely), but confirm the purpose's scope.

C6 "Suggest & confirm" (role-phrased)

In RAG → ask a question that includes the RAG-derived suggestion, phrased to the role; not in RAG → ask openly (then C7).

C7 User-supplied ingestion for unknowns

For anything not in RAG (incl. webhooks, FTP files): ask the user to supply the info or a doc link, then load it into RAG (Type 1a).

C8 Metric divergence is explained & recorded

When a new metric definition is not congruent with an existing one, ask the user why, then record the reason/explanation in RAG alongside the divergence.

C9 Two population phases

Initial load = generic (vendor + industry). Ongoing population = Q&A from users + telemetry as the agents work.

C10 Agents are tenant-specific

All agents are tenant-scoped; the tenant is the boundary. Global agents are deferred to a later spec.

C11 No canonicals on day one

Canonical (common) datamarts and patterns emerge from tenant usage. Industry seed provides generic semantics only; canonical status is earned, not seeded.

C12 Role ownership

Business = requirements · Analyst = definitions · Engineer = deployments. No additional owner roles in the current spec.

C13 Retrieval respects the embedding model version

Retrieval co-mixes only same-embed_model_version vectors; a model change implies a re-embed pass first.
