REFERENCE DOCUMENTATION · v1.0

Enterprise Semantic Data Assistant Architecture

A unified, business-language query layer across multiple enterprise systems — covering every metadata document, semantic catalog table, vector knowledge file, and runtime artifact used to build NLP pipelines with SQL, RAG, and Hybrid routing on top of a multi-source data estate.

Author: Anvizent Engineer Domains covered: HR · Finance · Sales · Inventory Source systems: 6 connected + 4 future Routing modes: SQL / RAG / HYBRID Pattern: Medallion + Semantic Layer + Governed RAG

01 Objective & Scope

The goal of this architecture is to give business users a single, conversational way to ask questions across every enterprise system the organization runs — without ever exposing physical tables, forms, columns, or source-system internals. A user asks a question in plain business language; the platform resolves that question down to physical data, retrieves supporting knowledge, and returns a business-friendly answer.

Example question: "Why did employee absences increase last month?"
The platform understands business intent → identifies business entities → resolves entities to source systems → retrieves facts from structured data → retrieves business knowledge from semantic repositories → generates a business-friendly response. None of this requires the user, or the prompting layer, to know that "Absence Date" is called ABS_DATE in Epicor and LEAVE_DT in Priority.

What this document covers

This is the canonical reference for every named document, table, and file the pipeline produces or consumes — at design time (building the semantic layer) and at run time (answering a live question) — with realistic example content for each, across multiple source systems and multiple business domains.

🛠️ Preparation Strategy: Manual vs. LLM-Assisted Mappings

Constructing and maintaining these column-to-attribute mappings at enterprise scale utilizes a Hybrid, Human-in-the-Loop workflow rather than purely manual or purely automated methods:

  • LLM-Assisted Bootstrapping (90% of efforts): When a new source system is onboarded, database DDLs, comments, and sample rows (3–5 records) are parsed by an LLM-based metadata crawler. The LLM evaluates syntax and value profiles to automatically suggest mappings to the standard business dictionary, recording its matching certainty in standardization_confidence.
  • Human Steward Verification (10% of efforts): To safeguard against schema hallucinations, proposed mappings are pushed to a staging queue. A human data steward verifies the mapping accuracy and updates the reviewed_by field with their signature. Only validated mappings are merged into the active standardized_attribute_map.json.
  • Drift Monitoring: Automatic alerts flag new source columns or altered data types, prompting LLM assessment and notifying the catalog stewards for final verification.

08 Layer 4 — Vector Knowledge Repository

Only business knowledge is vectorized — never raw transactional rows. Source documents are the glossary, KPI definitions, business rules, relationship explanations, and lineage summaries from Layer 3, chunked and embedded with metadata for filtered retrieval.

Document types stored

  • Business Glossary entries (vector_kb_glossary)
  • KPI / Metric definitions (vector_kb_kpi_definitions)
  • Business Rules & policy narratives (vector_kb_business_rules)
  • Relationship & lineage summaries (vector_kb_lineage)

Example embedded document

  sick leave and absence events. Sourced from Salesforce, Epicor and Priority ERP,
  conformed in DM_EMPLOYEE_ABSENCE. Contractor absences are included from March.",
  "sources": ["Salesforce", "Epicor", "Priority"],

  "vector_store": "pgvector",
  "approval_status": "governed",

Governance note: following the Self-Learning RAG pattern, the vector repository has two write paths — a governed path (catalog stewards approve new glossary/rule entries before embedding) and an auto-upgrade path (user-interaction-derived candidate knowledge queued for admin approval before upsert). The read-only Semantic Layer itself is never auto-modified.

Vector store options evaluated: pgvectorPineconeMilvus — selection depends on multi-tenant isolation requirements and existing Postgres footprint.

11 Universal Semantic Planner (LLM)

The Planner is the only component that talks to the raw user question. It is intentionally blind to physical structures — it understands business entities, metrics, time periods, relationships, and intent, and nothing else.

Understands

  • Business entities & metrics
  • Time periods & comparisons
  • Relationships between entities
  • Business intent (fact lookup, root cause, policy lookup, trend)

Never sees

  • Tables
  • Columns
  • Forms
  • Source system names

System prompt template (abridged)

You are the Universal Semantic Planner for an enterprise data assistant.
Given a business question, output ONLY a JSON object with:
  query_type, business_entities[], metrics[], time_period, filters{},
  requires_data, requires_explanation, requires_comparison, requires_trend_analysis.
You must reason using business_entities and the glossary provided in context.
NEVER reference table names, column names, or source system names.
If unsure of an entity, choose the closest glossary term and flag "entity_confidence": "low".

12 Semantic Resolver

The Resolver is the translation boundary between business language and physical metadata. It searches the Unified Semantic Catalog tables in a fixed order:

  1. semantic_entities — confirm the entity exists and its domain
  2. semantic_source_mappings — list every source system that has this entity
  3. semantic_metrics + semantic_metric_mappings — resolve metric to physical column per candidate source
  4. semantic_relationships — expand to parent/child entities if the question implies them (e.g. "by department")
  5. semantic_business_rules — attach any inclusion/exclusion logic that must travel with the metric
  6. semantic_glossary — expand synonyms back to the canonical entity name if the Planner's output was ambiguous

Source selection policy when multiple candidates exist

PriorityRule
1Prefer the Databricks Gold/Platinum data mart if the entity is conformed there (fastest, pre-aggregated, cross-source already reconciled)
2If not conformed, prefer the system marked as "source of truth" for that entity in semantic_source_mappings
3If a question explicitly names a system ("in Salesforce…"), honor that override
4If tenant-specific routing rules exist (multi-tenant deployments), apply tenant override before generic priority

The final, fully resolved instruction set handed to the SQL Agent. Always produced by the Resolver / source selector — never directly by the LLM, to keep execution deterministic and auditable.

15 SQL Agent

Generates parameterized, read-only SQL strictly from the Physical Execution Plan — never from free text — and always applies attached business rules as WHERE-clause predicates.

SELECT SUM(ABSENCE_COUNT)
FROM DM_EMPLOYEE_ABSENCE
WHERE MONTH = LAST_MONTH
  AND STATUS = 'APPROVED'; -- Approved Absence Only rule applied

Guardrails: read-only credentials, query templates (no dynamic table/column injection from user text), row-level tenant filters auto-appended, query cost/row limits enforced before execution.

16 RAG Agent

Performs dual retrieval (dense vector + keyword/sparse) against the Vector Knowledge Repository, filtered by the entity and domain resolved earlier, with a minimum similarity threshold and a small top-k.

// rag_agent_output.json
{ "search_terms": ["Employee Absence","Absence Count","Policy Changes","Leave Rules"],
  "top_k": 3, "similarity_threshold": 0.74,
  "documents": [
    "Contractor absences included from March",
    "New leave categories introduced",
    "Approved absence rule updated" ] }

17 Context Builder

Merges the original question, SQL facts, and RAG knowledge into one bundle, and runs a lightweight consistency check (e.g. does retrieved knowledge actually explain the direction of the SQL fact?) before handing off.

{
  "question": "Why did employee absences increase last month?",
  "facts": { "current_month": 124, "previous_month": 98, "increase_percent": 26.5 },
  "knowledge": ["Contractor absences included", "New leave categories introduced"],
  "consistency_check": "passed"
}

18 Answer Generation LLM

Takes Question + Facts + Knowledge and produces the final business-friendly narrative — under a strict instruction never to mention table or column names.

Employee absences increased by 26.5% compared to the previous month.
The increase appears to be associated with:
• Inclusion of contractor absences.
• Introduction of new leave categories.
• Updated absence counting rules.

19 New Connector Onboarding Checklist

Per the core design principle, adding a system like Workday or Oracle EBS never touches the Planner, Resolver logic, or Routing Engine — only metadata onboarding:

  • Crawl source metadata into a new <prefix>_tables / <prefix>_columns repository (Layer 1)
  • Add crosswalk rows to standardized_attribute_map.json (Layer 2)
  • Add or extend rows in semantic_source_mappings and semantic_metric_mappings (Layer 3)
  • Review/extend semantic_business_rules if the new source has different inclusion logic
  • Re-embed any new/changed glossary or rule text into the Vector Knowledge Repository (Layer 4)
  • Run regression scenarios (Section 10 style) against the new source before enabling it for routing
  • Set source priority in the Resolver's selection policy if the new source competes with an existing one

21 Design Principles

  • Source metadata remains source-specific — no renaming at Layer 1.
  • The semantic catalog remains business-centric and is the only layer the Planner reasons over.
  • The Planner understands business language only — never tables, columns, or source systems.
  • The Resolver is the sole translator from business language to physical metadata.
  • SQL produces facts; RAG produces knowledge; the Answer LLM produces insight — each agent has exactly one job.
  • New connectors require metadata onboarding only — the Planner and routing logic never change.
  • Business users never see physical tables or columns, in the question or in the answer.
  • The semantic catalog is versioned and governed — changes go through steward approval, not silent auto-write.
  • The vector repository has a governed path and a separately gated auto-upgrade path — never direct write from user interactions.
  • Every runtime artifact (*_plan.json, *_decision.json) is logged for auditability and reproducibility of an answer.
  • Multi-tenant deployments apply tenant isolation at the Resolver's source-selection step, not as an afterthought in SQL.
