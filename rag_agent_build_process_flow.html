Cross-cutting invariants

These are not phases — they wrap every identification step in the flow below.

1 · "Suggest & confirm" RAG pattern

For every element the agent needs to identify:

  • In RAG  → ask with the RAG-derived suggestion ("I plan to use … — confirm or correct?"), phrased to the role.
  • Not in RAG  → ask openly. If the user doesn't have it either, ask them to supply it or a doc link, then load it into RAG.

2 · Role-adaptive questioning & ownership

The same decision is phrased — and the confirmation routed — to the role's ownership:

Business = owner of requirements · Analyst = owner of definitions · Engineer = owner of deployments.

Reconciliation & extraction hygiene enforced at JSON validation

Applied throughout the build and checked at validation:

  • Extract by date/time range only — never by business attributes.
  • Extract each dataset individually; never pre-join during extraction.
  • Row-level uniqueness on join keys before any join — dedup, null defaults, cleanse.
  • Cross-reference every dimension; flag unmatched as "Unknown" — never drop records.
Tenant scope. The agent is tenant-specific — all retrieval and production stay within the tenant. Global agents are deferred.
The ordered method

Process flow — 19 steps

Each step carries the invariants above: suggest & confirm against RAG (ask openly + ingest if missing), phrase to the role's ownership, and hold the reconciliation/extraction hygiene.

Scope
1

Intake & source applications

Get the business question. Identify the source applications.

Ask if not in RAG; if present, state the systems you'll use to fetch and invite dispute.
2

Scope all sources (clarify the metrics in the question)

Across all source applications, identify:

  • tables / APIs from each application
  • webhooks
  • files present in any FTP locations, etc.
3

Elements per source

For each identified source, identify the elements (fields) needed.

Same pattern — ask if not in RAG, confirm if it is.
Warehouse layer
4

DW table classification & dimension resolution

These are the DW tables. Identify which are transaction tables and which are dimension tables. Bring all related dimension resolution for every transaction table. Identify what data to filter based on how far back in history you need.

Same RAG approach (ask / confirm).
5

Load strategy

Identify whether you need a full load or an incremental load.

6

DW table JSON

Create JSON for DW tables that are not already there. If they already exist, add only the columns that are needed.

Datamart construction
7

Decompose & reuse check sub-datamarts

Break the requirement into sub-datamarts. Check the datamart catalog for existing datamarts/sub-datamarts that already answer a sub-component; reuse and compose them, adding only net-new DW tables/columns rather than rebuilding.

8

Driving table

Identify the driving table — the primary fact / transaction table that anchors the analysis.

9

Filter / group-by order

Identify filter → group by or group by → filter.

10

Filter & group-by specifics

  • columns / fields in the filter and in the group by
  • values and conditions in the filters
  • columns in the group by
11

Transaction-table joins

Identify if you need to join with other transaction tables.

12

Lookup tables

For each lookup table, identify whether you need to: filter & group by · group by & filter · just group by · just filter — and the corresponding values.

13

Join conditions role-adaptive

Identify the join conditions from RAG (suggest & confirm). Ask per role — business user keeps the business implication; analyst & engineer get table- and column-level.

14

Field selection & renaming

Keep the fields you need and drop the ones you do not. Rename fields to make business sense based on context.

15

Calculated values / metrics

Identify the metrics needed. First build the metrics common across more compound metrics to keep the formulae consistent, then the rest.

Build all metrics by asking + RAG. On a definition incongruent with an existing one, ask why and record the reason in RAG — never diverge silently.
16

Final shaping

Filter & group by, or group by & filter, with the relevant elements if needed.

Finalize
17

Scope, usage & ownership confirmation

Elicit the result's scope (company / department / purpose / role / user). Confirm the usage — onetime (low reliance, narrow) or scheduled (a purpose served; rely, but confirm its scope). Route confirmations to owners: requirements → business, definitions → analyst, deployment → engineer.

18

JSON validation

Validate the JSON, enforcing the reconciliation & extraction-hygiene rules (Unknown-flagging, row-level uniqueness before joins, date-range extraction, no pre-join).

19

Emit & record

Emit the datamart JSON. On confirmation + validation + usage, record it as Type 6 resolved contextual semantics (vector + datamart catalog), scope-tagged and linked to its sub-datamarts; the generalized pattern feeds Type 2 and earns canonical status only through repeated usage.

Step 19 output

Shape of the datamart JSON

Illustrative only — the structure mirrors the build steps (scope, driving table, joins, lookups, filter/group-by order, field selection, renames, metrics) and is recorded as Type 6.

// illustrative datamart spec — shape follows steps 7–19
{
  "scope": "department",                  // step 17 (company..user)
  "usage": "scheduled",                  // step 17 (onetime|scheduled)
  "composes_of": ["sub_dm_headcount"],   // step 7 reusable sub-datamarts
  "load": "incremental",                 // step 5
  "driving_table": "fct_absences",        // step 8
  "order": "filter_then_groupby",         // step 9
  "filters": [                            // step 10 (date range = step 4)
    { "field": "absence_date", "op": ">=", "value": "2025-01-01" }
  ],
  "group_by": ["department_id", "region"],
  "transaction_joins": [],                 // step 11
  "lookups": [                            // step 12
    { "table": "dim_employee", "mode": "group_by_only" },
    { "table": "dim_department", "mode": "filter_then_group_by" }
  ],
  "joins": [                              // step 13 (from RAG)
    { "left": "fct_absences.employee_id",
      "right": "dim_employee.employee_id",
      "type": "left", "unmatched": "Unknown" }  // step 18 hygiene
  ],
  "fields": [                             // step 14 keep + rename
    { "src": "dim_department.dept_name", "as": "Department" }
  ],
  "metrics": [                            // step 15 base first, then compound
    { "name": "Absence Hours", "expr": "SUM(absence_hours)" },
    { "name": "Absence Rate", "expr": "Absence Hours / Scheduled Hours" }
  ]
}
After the build

Datamart reuse evaluation (compositional)

Reuse is compositional: decompose the requirement into sub-components, where each sub-component can be a datamart. Each sub-datamart can be composed into the current final datamart and reused in other datamarts for other questions.

Reuse over rebuild. Decompose → match sub-components against the datamart catalog → reuse what exists, adding only new DW tables/columns → compose the final mart. Canonical (common) datamarts are earned through tenant usage, not seeded.
Where this sits

Ties to the RAG

Consumes the RAG

Every "is it in RAG?" check reads the knowledge store: structural docs (Type 1a), live schema (Type 1b cache), prior patterns (Type 2), semantics & metrics (Type 3), the tenant's customizations (Type 4), and resolved datamarts (Type 6) — so the agent can suggest what it will use and let the user confirm. Retrieval respects the active embed_model_version.

Produces for the RAG

On confirmation + validation + usage (not deploy): the datamart JSON is recorded as Type 6 resolved semantics (+ the generalized Type 2 pattern, canonical only via repeated use); metric definitions queue as Type 3 suggestions (divergences recorded with a reason); user overrides become Type 4; run telemetry distills to Type 5.
