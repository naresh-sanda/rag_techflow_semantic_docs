Internal Architecture Brief — Cleaned & Expanded

Enterprise Semantic Data Assistant Architecture

A single AI assistant that lets anyone ask business questions in plain English — across Salesforce, Epicor ERP, Priority ERP, and Databricks — without ever knowing a table name, a column name, or which system holds the answer.

Core Question Answered"Why did employee absences increase last month?"

Orientation

What this system actually does

Before any layer or diagram, here is the one-sentence version, in plain language, of the whole document.

In one sentence A user types a business question → the system figures out what business thing they mean, quietly decides which backend system actually holds that data, pulls the numbers, pulls relevant business context, and hands back a written answer — never showing the user a table name, SQL query, or system name.

The platform unifies query access across five categories of enterprise systems:

CRM
Salesforce
  • Custom objects & forms
  • e.g. ABSENTCHART
ERP
Epicor ERP
  • Relational tables
  • e.g. HR_ABSENCE
ERP
Priority ERP
  • Relational tables
  • e.g. EMP_ABS
Lakehouse
Databricks Data Warehouse
  • Curated datamarts
  • e.g. DM_EMPLOYEE_ABSENCE
Extensible
Future Connectors
  • Onboarded as metadata, not code
  • SAP, HubSpot, Snowflake, etc.

The end-to-end promise the architecture makes, broken into its five constituent capabilities:

  1. Understand the business question (not the database schema).
  2. Identify the business entities involved — e.g. "Employee Absence."
  3. Resolve those entities to whichever source system actually has the best data.
  4. Retrieve facts from structured data (numbers, counts, trends).
  5. Retrieve knowledge from semantic/business repositories (definitions, rules, "why" context) — then generate one business-friendly response.
Orientation

The mental model: two phases, not ten layers

The original document lists 5 architecture layers and 6+ runtime steps as one long list, which is where most of the confusion comes from. They are actually two completely separate phases that run at different times. Keep this split in your head and everything else clicks into place.

Phase A — Build Time
"Teach the system the business" (Layers 1–5)

Done once, then maintained incrementally. Engineers and metadata pipelines crawl every source system, normalize the findings, and write them into a catalog and a vector store. No end user is involved in this phase. This is the "library being stocked and catalogued" before anyone walks in to ask a question.

Phase B — Run Time
"Answer the question" (Steps 1–6 + synthesis)

Happens every single time a user types a question — typically in 1–3 seconds. It reads everything Phase A built; it never writes new metadata. This is the "librarian" who hears your question and knows exactly which shelf, which book, and which page to open.

Why this split matters for implementation Phase A is a metadata/data-engineering problem (crawlers, normalization jobs, embedding pipelines). Phase B is an LLM-orchestration problem (prompt design, routing logic, API calls). They can be built by different sub-teams, on different timelines, and Phase B literally cannot produce a correct answer until Phase A has populated the catalog for the relevant business entity.
Phase A · Build Time — Layer 1

Source Metadata Repositories

Purpose: store each source system's metadata exactly as that system describes itself — no translation, no cleanup yet. This is the raw intake.

Why keep it "as discovered"? If you normalize too early, you lose the ability to trace a business answer back to its real source field for debugging or audit. Layer 1 is intentionally messy and source-specific — cleanup happens one layer later, in Layer 2.

Example: the same business fact, stored three different ways

SourceContainerColumns (as-is)
SalesforceABSENTCHART (Object)EMP_ID, ABSENCE_DATE, ABSENCE_REASON
EpicorHR_ABSENCE (Table)EMPLOYEE_NUM, ABS_DATE, ABSENCE_CODE
PriorityEMP_ABS (Table)EMP_NO, LEAVE_DT, REASON_CODE

Notice: three systems, three naming conventions, one underlying business idea — "when did this employee miss work, and why." That gap is exactly what Layer 2 closes.

Phase A · Build Time — Layer 2

Metadata Standardization

Purpose: normalize the messy, source-specific metadata from Layer 1 into one consistent naming and typing convention, while keeping a record of which raw field each standardized name came from.

The six normalization activities

Normalize names

ABS_DATE, LEAVE_DT → one shared name

Normalize datatypes

e.g. text date vs. date type → unified type

Normalize descriptions

Inconsistent field labels → one definition

Extract aliases

All known names for the same concept

Extract synonyms

Business-language variants users might type

Standardize relationships

Foreign keys → one relationship model

Worked example

SourceRaw field name
SalesforceABSENCE_DATE
EpicorABS_DATE
PriorityLEAVE_DT
Standardized output All three collapse into a single Standardized Business Attribute: "Absence Date." Anywhere downstream in this architecture says "Absence Date," it transparently means any of those three raw columns, depending on which source is ultimately selected.
How Standardized Maps are Prepared (Manual vs. LLM)

Building these mappings at scale is performed via a Hybrid, Human-in-the-Loop workflow:

  • LLM-Assisted Bootstrapping (90% of effort): Database DDLs, column metadata, and sample rows (3–5 records) are parsed automatically by LLM agents. The LLM suggests mappings to standard business dictionary terms and tags its certainty in a standardization_confidence property.
  • Data Steward Verification (10% of effort): Proposed mappings are queued for verification. A human data steward checks the matches for accuracy and signs off by writing their identifier to the reviewed_by field, preventing hallucinations from entering active routing configurations.
  • Schema Evolution: Any automated table sync or column modification flags changes, rerunning LLM mapping suggestions for steward confirmation.
Phase A · Build Time — Layer 3

Unified Semantic Catalog

Purpose: the central knowledge layer of the whole platform — a business-centric view of everything discovered and standardized so far. This is the single source every later runtime step queries against.

Think of it as A business dictionary + an org chart + a "where do I find this" index, all in one catalog — built for business terms, not table names.

Knowledge Generation

Purpose: take the structured catalog from Layer 3 and generate readable, narrative business knowledge from it — text that an LLM can later retrieve and quote from when a user asks a "what is / why" question rather than a "how many" question.

Phase A · Build Time — Layer 5

Vector Knowledge Base

Purpose: take the narrative text from Layer 4 and embed it into a vector store so it becomes searchable by meaning, not just keyword. This is what powers the RAG route at runtime.

What's stored as vectors

  • Business Rules
  • Relationship Descriptions
  • KPI Definitions
  • Lineage Summaries

Why this matters at runtime When a user later asks something conceptual like "What is Employee Absence?", the system doesn't run SQL — it does a similarity search against this vector store and returns the closest matching definition. See RAG Route below.

Runtime · Step 1

User Question

The only input the end user ever provides — no forms, no filters, no system selection.

Input

Why did employee absences increase last month?
Runtime · Step 2

Universal Semantic Query Planner

Purpose: convert the natural-language question into a structured Business Query Plan — and critically, do this with zero knowledge of any source system. The planner only ever thinks in business terms.

Its five responsibilities

Spacer for details
Entity Detection

"employee absences" → Employee Absence

Metric Detection

"increase" implies a count/measure: Absence Count

Time Detection

"last month" → a resolved date range

Filter Detection

Any qualifiers ("for contractors", "in Sales")

Input

Why did employee absences increase last month?

Output — Business Query Plan

{
  "query_type": "root_cause",
  "business_entities": ["Employee Absence"],
  "metrics": ["Absence Count"],
  "time_period": "Last Month",
  "requires_data": true,
  "requires_explanation": true
}
Why "no source knowledge required" matters Because the planner never mentions Salesforce, Epicor, or anything physical, you can add a brand new connector tomorrow and this component needs zero changes. All source-awareness is deliberately pushed downstream, to Steps 3–5.
Runtime · Step 3

Unified Semantic Catalog Lookup

Purpose: take the business entity from Step 2 and ask the Layer 3 catalog, "which systems even have this?" This step only narrows down candidates — it does not yet decide the winner.

Input

Employee Absence

Output

{
  "candidate_sources": [
    "salesforce", "epicor", "priority", "databricks"
  ]
}

This is a direct lookup against the Source Mappings table you saw in Layer 3 — nothing is computed here, it's pure retrieval.

Runtime · Step 4

Semantic Resolver

Purpose: out of the candidate list from Step 3, pick the single best source to actually query. This is the component that embodies "prefer the clean curated data over the messy operational system."

Decision rules, in priority order

  • 1
    Prefer Datamarts if availableA curated Databricks datamart beats querying a live operational system directly.
  • 2
    Prefer curated datasetsCleaned, deduplicated, business-ready data wins over raw source tables.
  • 3
    Use source systems only when requirede.g. when no datamart exists yet for this entity, or real-time freshness is demanded.
  • 4
    Use multiple systems when necessarye.g. a question spanning both CRM and ERP data simultaneously.

    • Output

    {
  "selected_source": "databricks",
  "business_entity": "Employee Absence"
}

    Databricks wins here because Rule 1 fires immediately — DM_EMPLOYEE_ABSENCE is exactly the kind of curated datamart this rule prefers.

    Runtime · Step 5

    Physical Execution Plan

    Purpose: this is the moment the system finally drops from "business language" down to "physical metadata" — real table names, real column names. Everything before this step was abstract; everything after is concrete and executable.

    Output

    {
  "source": "databricks",
  "table": "DM_EMPLOYEE_ABSENCE",
  "metric_column": "ABSENCE_COUNT",
  "date_column": "ABSENCE_DATE"
}

    From this point forward, the system has full operational knowledge: source, table, columns, and relationships — everything needed to actually go fetch data.

    Runtime · Step 6

    Routing Engine

    Purpose: this is the fork in the road. Using the two booleans the Planner set back in Step 2 (requires_data, requires_explanation), the engine decides which of three execution paths to take.

    requires_datarequires_explanationRouteMeaning
    truefalseSQLPure number lookup — "how many"
    falsetrueRAGPure concept lookup — "what is"
    truetrueHYBRIDNeeds numbers and a narrative — "why did"
    The simplest way to remember this SQL = facts only. RAG = knowledge only. Hybrid = facts + knowledge merged by an LLM into one answer. Three named scenarios below walk through each route end-to-end with a real example.
    Scenario 1 of 3

    SQL Route — pure facts

    Fires when the user wants a number and nothing else — no narrative needed.

    SQL Route

    requires_data=true · requires_explanation=false

    Example question

    How many employee absences occurred last month?

    Generated SQL (built from the Step 5 physical plan)

    SELECT COUNT(*) FROM DM_EMPLOYEE_ABSENCE
WHERE ABSENCE_DATE BETWEEN '2026-05-01' AND '2026-05-31';

    Output

    { "absence_count": 124 }

    Result type

    FACTS

    The user receives this number directly (optionally lightly phrased), with no detour through the vector store at all.

    Scenario 2 of 3

    RAG Route — pure knowledge

    Fires when the user wants a definition, rule, or explanation — no database query needed at all.

    RAG Route

    requires_data=false · requires_explanation=true

    Example question

    What is Employee Absence?

    Vector search target

    Employee Absence Definition

    Output

    {
  "knowledge": [
    "Employee absence records track leave, vacation and sick leave."
  ]
}

    Result type

    KNOWLEDGE

    This is a similarity search against the Layer 5 Vector Knowledge Base — Steps 3–5's table/column resolution is skipped entirely, because there's no number to fetch.

    Scenario 3 of 3

    Hybrid Route — facts + knowledge merged

    The flagship scenario of this entire architecture — the original "why did absences increase" question. Needs a number and a reason, so both the SQL and RAG paths fire in parallel, then get merged.

    Hybrid Route

    requires_data=true · requires_explanation=true

    Example question

    Why did employee absences increase last month?
    SQL side

  "current_month": 124,

    merged by the Context Builder, below
    Runtime · Synthesis, Stage 1

    Context Builder

    Purpose: only relevant in the Hybrid route. Combines the original question, the SQL facts, and the RAG knowledge into one structured payload, ready to hand to the final LLM.

    Combines

    • User Question
    • SQL Facts
    • RAG Knowledge

    Output

    {
  "question": "Why did employee absences increase last month?",

    Runtime · Synthesis, Stage 2 — Final Step

    Answer Generation LLM

    Purpose: the last component in the chain. Takes the merged context and writes the actual sentence the user reads — turning structured JSON into natural business prose.

    Input

    Question + Facts + Knowledge

    Final output shown to the user

    • Inclusion of contractor absences

    Synthesis

    Key design principles

    The eight rules that, taken together, are the actual "architecture" — everything above is just these rules made concrete.

    • 1
      Planner understands business language.It never touches a table name (Step 2).
    • 2
      Semantic Catalog understands business concepts.It's the dictionary every other step queries (Layer 3 / Step 3).
    • 3
      Resolver understands source systems.All "which backend wins" logic lives in exactly one place (Step 4).
    • 4
      SQL produces facts.Numbers, counts, trends — never narrative (SQL Route).
    • 5
      RAG produces knowledge.Definitions, rules, context — never raw numbers (RAG Route).
    • 6
      Final LLM produces insights.It's the only component allowed to blend facts and knowledge into prose.
    • 7
      Source systems remain hidden from end users.The user experience is "ask a question," never "pick a system."
    • 8
      New connectors require metadata onboarding, not planner changes.Adding SAP tomorrow only touches Layers 1–3 — Steps 2, 5, and 6 don't change at all.
    Synthesis

    Clarity checkpoints — questions this kind of architecture usually raises

    Quick answers to the questions that typically come up once an architect actually sits down to build this.

    Is this a single application, or several separate services?
    In a real implementation this is several services: a metadata-crawling pipeline (Layer 1–2, scheduled jobs), a catalog service (Layer 3, typically backed by something like a metadata store or even a well-modeled relational DB), an embedding/indexing job (Layer 4–5, feeding a vector DB such as pgvector / Pinecone / Milvus), and a runtime orchestration service (Steps 1–6 + synthesis, typically an LLM-orchestration layer calling tools/functions for catalog lookup, SQL execution, and vector search).
    Where does the actual SQL get executed?
    Against whichever physical system the Semantic Resolver (Step 4) picked — in the running example, Databricks. The Physical Execution Plan (Step 5) is what gives the SQL Route the concrete table/column names it needs to build a real, executable query.
    What happens if the Semantic Resolver picks a source that doesn't have a column the question needs?
    This document doesn't specify a fallback explicitly, but the implication of Rule 4 ("use multiple systems when necessary") is that the resolver can select more than one source and the routing/execution layer fans out across both, merging results before they reach the Context Builder.
    How is "last month" turned into real start/end dates?
    That's part of Time Detection inside the Query Planner (Step 2) — it resolves relative time phrases against the current system date before the plan is handed downstream, so by the time SQL is generated in the SQL Route, it's already a concrete date range.
    Does every question go through all six steps?
    No. Pure RAG questions (e.g. "What is Employee Absence?") skip the Physical Execution Plan and SQL generation entirely, since there's no data to fetch — they go straight from Catalog Lookup to a vector search. Pure SQL questions skip the vector store entirely. Only Hybrid questions touch every component.
    How do you add a brand-new source system, like SAP, later?
    Per Principle 8: you onboard SAP's metadata into Layer 1, run it through the Layer 2 standardization activities, and add its entries to the Layer 3 Source Mappings table. Nothing in Steps 2 (Planner), 5, or 6 (Routing) needs to change — the Resolver (Step 4) will simply see SAP as one more candidate source the next time someone asks about an entity SAP happens to also hold.
    Suggested Next Step

    A practical, phased implementation plan

    A suggested sequencing to actually build this, mapped directly onto the layers and steps above — useful as a starting backlog, not a fixed prescription.

    01

    Stand up the Source Metadata Repositories (Layer 1)

    Build a metadata crawler per source — start with one system end-to-end (e.g. Databricks, since it's already curated) before adding Salesforce/Epicor/Priority.
    02

    Build the Standardization layer (Layer 2)

    Write the name/datatype/alias normalization rules. This is where a human data steward typically reviews and approves mappings before they go live.
    03

    Stand up the Unified Semantic Catalog (Layer 3)

    Pick a concrete store for this — a metadata/catalog database with entities, attributes, relationships, glossary, and source-mapping tables, queryable by both the Resolver and a human admin UI.
    04

    Generate Knowledge and embed it (Layers 4–5)

    Run an LLM batch job that reads catalog entries and writes glossary/rule/KPI narratives, then embed those narratives into a vector DB (pgvector, Pinecone, or Milvus).
    05

    Build the runtime Query Planner (Step 2)

    An LLM-backed component, prompted to emit exactly the Business Query Plan JSON shape shown above — test it against 20–30 representative business questions before wiring up downstream steps.
    06

    Wire Catalog Lookup → Resolver → Physical Plan (Steps 3–5)

    Mostly deterministic lookups and rule evaluation against Layer 3 — lower LLM-risk, good to build and unit-test early.
    07

    Implement the Routing Engine and all three routes (Step 6)

    Start with the SQL Route alone (simplest), then RAG Route, then Hybrid last, since Hybrid depends on both being correct first.
    08

    Add the Context Builder and Answer Generation LLM

    Prompt-engineer the final synthesis step carefully — this is the only component the user actually sees, so its tone and accuracy matter most.
    09

    Onboard the second and third source systems

    Once one full vertical slice works end-to-end for Databricks, repeat Layer 1–3 onboarding for Salesforce, Epicor, and Priority — this validates Principle 8 in practice.
