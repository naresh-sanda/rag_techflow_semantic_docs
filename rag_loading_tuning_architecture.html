How knowledge enters, refreshes, and exits the RAG

The concrete, locked design for creating and maintaining the Anvizent RAG: five knowledge types across three loading regimes, a Postgres + pgvector store, first-class eviction, and a configuration layer that tunes itself from telemetry — added to an existing RAG without rebuilding it.

Scope & Principle

An add-on, not a rebuild

The existing RAG is treated as a substrate that stays almost untouched. The add-on attaches through three seams and otherwise runs out of band — so it can never take the core RAG down.

Design principle: minimal-touch, fail-open

The add-on requires exactly two code changes inside the existing RAG; everything else wraps the write path or runs as out-of-band workers.

Fail-open guarantee. If the control plane is unavailable, the resolver returns last-known / default values and retrieval keeps working. The add-on can degrade, but it cannot break the core RAG.

What it adds

  • Loading orchestration — the right regime for each knowledge type
  • Eviction / hygiene — the index can't fill with stale, confidently-retrieved knowledge
  • Configuration — every calibration knob is a live, audited config record
  • Adaptive tuning — a controller moves knobs within bounds from telemetry
Organizing principle

Three loading regimes

The recurring failure mode is conflating these. Hot loops optimize for latency and auto-write; curated loads optimize for review and versioning; live retrieval must never be embedded at all, or the index rots as schemas drift.

RegimeLatencyWrite pathFailure costTypes
Hot / event-driven Real-time to seconds Automatic on event Self-correcting (decays out) Patterns, Customizations
Gated / curated Release cadence Human-reviewed High — poisons every downstream generation Semantics, Structural-doc corpus
Live retrieval Query-time No write to vector store Stale snapshot Live source schema
The five knowledge types

What gets loaded, and how

Each type names its source, trigger, transform, namespace, refresh, and — critically — its eviction rule. Select a type to inspect its loading contract.

TYPE 1aStructural doc corpusGated · indexed
TYPE 1bLive source schemaLive · never indexed
TYPE 2Integration patternsHot · self-improving
TYPE 3Business semanticsGated · human-curated
TYPE 4Customer customizationsHot · tenant-scoped
TYPE 5Telemetry feedbackHot · platform-generated

Type 1a — Source-system structural knowledge Gated

The indexed half of Type 1. Split from the live track because the live/static distinction is what keeps the index from rotting.

Source
Vendor data dictionaries, OpenAPI / API specs, published ERDs, internal connector docs.
Trigger
SOR onboarding; vendor structural version release.
Transform
Parse → section-aware chunk → embed → tag (system, version, object_type).
Namespace
kb:sor-docs:{system}:{version}
Refresh
On vendor version bump; periodic audit for drift between docs and observed reality.
Eviction
Version-supersede. Prior version deprecated, retained while any pipeline is pinned to it, hard-deleted once no pins remain.

Type 1b — Live source schema Never indexed

Held as structured, cached metadata — never written into the vector store. Embedding it would rot the index as schemas drift.

Source
Connector introspection via the metadata service.
Trigger
Query-time assembly; schema-drift webhook.
Transform
None into the vector store — held as structured, cached metadata.
Namespace
Metadata-service cache, not the vector index.
Refresh
Short TTL plus drift-triggered invalidation.
Eviction
Cache expiry; a drift event invalidates immediately.

Type 2 — Integration & reconciliation patterns Hot

The self-improving loop. Has to run hot. Tenant data must be redacted before promotion to the global namespace.

Source
Pipeline/datamart JSON produced by the agent and confirmed-and-used; spec-library commits. (Concrete runnable JSON lives in Type 6.)
Trigger
Usage-based, not deploy-based. Confirmation + data validation + usage — onetime = low reliance/narrow; scheduled = a purpose served (rely, confirm scope).
Transform
Extract pattern signature (source pair, transform graph, reconciliation rules) → scrub tenant-specific values → generalize → embed → tag (industry, source_systems, pattern_class).
Namespace
kb:patterns:global
Canonical
Earned, never seeded. No canonicals on day one — a pattern becomes canonical only once it recurs across tenant usage.
Eviction
Combined decay — deprecated when failure rate crosses the cutoff or selection count over the rolling window drops to disuse; explicit deprecate on breaking schema change.

Type 3 — Business semantics & metrics Gated

Powers the metrics-suggestion agent. Must stay human-gated so the agent never manufactures consensus from a handful of tenants.

Source
Curated industry glossaries authored by the team; seeded from industry docs; suggestions surfaced from observed customer metric definitions.
Trigger
Gated curated version release; vertical onboarding.
Transform
Author → review/approve → version → embed with definition, formula, industry tag, synonyms. Customer-surfaced definitions accumulate in a batch-proposed-diff queue reviewed as a set.
Namespace
kb:semantics:{industry}:{version}
Refresh
Periodic, human-approved releases; suggestion queue reviewed in batches.
Eviction
Version-supersede. Corrected definitions replace prior; prior retained read-only for lineage.
Guardrail. Observed customer definitions may suggest additions but never auto-write into the canonical glossary.

Type 4 — Customer customizations Hot · tenant-scoped

Isolation is non-negotiable. Physical database-level isolation, not a query-time filter — retrieval cannot reach across tenants by construction.

Source
Customer overrides to generated artifacts — renames, custom business rules, transform tweaks.
Trigger
Save / commit on a customer artifact.
Transform
Diff against generated baseline → capture delta → embed → tenant-tag.
Namespace
Dedicated Postgres database per tenant; customizations live in a custom table within the tenant DB.
Refresh
Real-time on save.
Eviction
Revert removes the entry; re-edit supersedes; tenant offboarding is a single DROP DATABASE.

Type 5 — Operational telemetry feedback Hot

The only type generated by the platform itself. Never load raw events — distill first.

Source
Processing-engine run telemetry — success/failure, performance, data-quality outcomes, bias-detection results.
Trigger
Micro-batch window close (rolling, hourly to daily).
Transform
Hybrid (rules → LLM). Rules do the deterministic gate — aggregate, threshold, filter to actual signal — then an LLM phrases only the survivors into condition → outcome insight statements. The LLM never sees runs that didn't clear the gate.
Namespace
Global insights in kb:telemetry-insights:global; tenant-specific failure modes in the tenant DB.
Refresh
Micro-batch on a rolling window.
Eviction
Confidence / time decay — insights age out as conditions change; superseded when contradicted by a newer aggregate.

Type 6 — Resolved contextual semantics DW / datamart catalog

The data-warehouse / datamart layer the agent operates on. A datamart's working JSON + its DW tables + the columns it uses are themselves semantic knowledge — the resolved context for a question, reusable across the company.

Source
The agent's confirmed-and-used datamart / DW-table JSON (the agent is Source C).
Trigger
Usage-based (same as Type 2): confirmation + data validation + usage — not deploy.
Transform
Discovery vector in kb_chunk (resolved_semantics) plus the concrete runnable JSON in relational datamart_catalog / dw_table_catalog.
Scope
Required. company · department · purpose · role · user — elicited by the agent and tagged on each record.
Composition
composes_of links — a requirement decomposes into reusable sub-datamarts usable here and in other datamarts.
Namespace
Tenant DB (agents are tenant-specific). kb:resolved:{scope}:{ref}
Eviction
Supersede on re-confirmation; disuse decay. Canonical (common) datamarts are earned via usage, not seeded.
Cross-cutting

Shared infrastructure & locked decisions

Shared infrastructure

  • Embedding-model versioning. Stamp the model version into every record; changing the model means a re-embed pass. Mismatched-model vectors retrieved together degrade silently.
  • Common metadata schema. Every record carries type, source, tenant_scope, embed_model_version, created_at, supersedes.
  • Tenant isolation. Database-per-tenant for Type 4 + tenant telemetry; redaction-before-promotion for Type 2 + global telemetry; global-by-construction for Types 1a and 3.
  • Eviction as first-class. Every type names a supersede/decay rule — without it the index fills with confidently-retrieved stale knowledge.

Locked decisions

  • Store: Postgres + pgvector — vectors next to relational metadata, no separate infra.
  • Type 4 isolation: Database-per-tenant; DROP DATABASE offboarding; tenant→DB control-plane catalog.
  • Type 5 distillation: Hybrid — rules gate, LLM phrases survivors.
  • Type 5 scope: Both global and tenant insights; tenant→global requires scrub + generalize.
  • Type 2 decay: Combined failure-rate and disuse triggers.
  • Type 3 intake: Batch-proposed-diff review queue.
  • Type 6: Resolved contextual semantics — DW/datamart catalog (vector + relational), scope-tagged & composable.
  • Capture is usage-based, not deploy-based: confirmation + validation + usage (onetime = narrow/low; scheduled = purpose served).
  • Canonicals are emergent, not seeded: earned through tenant usage.
  • Agents are tenant-specific: global agents deferred.
Configuration & adaptive tuning

Calibration knobs are first-class config

Not deferred decisions — every tunable is a config record with two control paths: manual override by advanced users and telemetry-driven auto-tuning by a controller that reuses the Type 5 distillation stream. The same loop that feeds RAG insights also feeds the tuner.

Control: one check per parameter

The user-facing control is a single toggle per parameter — Locked (manual) or Learning-based — not three modes to reason about. Whether a learning-based change applies on its own or waits for sign-off is the parameter's own apply_behavior, not a third choice the user makes.

Retrieval top-k
Learning-based · auto-apply
Embedding model
Learning-based · propose-for-approval (re-embed)

Toggle a switch to see how the same one check produces correct blast-radius behavior underneath.

Precedence & scope

user lock > learned value > default

Locking a parameter (unchecking learning) pins it; checking learning hands control back to the controller within bounds. Scope is global by default; per-tenant overrides live in the tenant's own database, so a tenant with unusual data can be tuned — or locked — without touching the fleet.

The tuning controller (reuses Type 5)

  1. Compares the live objective against target.
  2. Proposes a bounded delta.
  3. Shadow- or A/B-evaluates before fleet-wide apply.
  4. Auto-rolls-back if the objective regresses.
  5. Never touches locked params; never exceeds bounds; rate-limited per interval.
Auditability. Every change is a versioned config event with who/what changed it and the justifying_window behind a learned change. You can always answer "why is top-k 11 right now" and roll back.
Implementation architecture

Three seams, one brain

The control plane is new; the existing RAG is reused. The add-on attaches through a write seam, a read seam, and a telemetry seam — and the read seam is the only place existing code changes.

The read seam is mechanical: swap constants for resolve(...), wrapped in a small middleware so it's one edit, not many.

Runtime flows

Ingest

Source event → Loading Orchestrator → type adapter transform (scrub / distill / chunk) → metadata stamp → existing ingestion API → pgvector. Live schema (1b) skips the store.

Retrieve

Query → retriever calls resolver.resolve(top_k|threshold|reranker, scope) → executes against pgvector (tenant DB if tenant-scoped) → generator.

Tune

Generator emits outcome → Collector → distillation → Controller compares objective to target → bounded proposal → (auto) write current_value + audit, or (propose) queue for Admin → Resolver cache invalidated.

Evict

Hygiene worker scans metadata → applies supersede / decay rules → deprecates or deletes vectors.

Net-new vs. reused

Reused (no internal change)

Embedding service, pgvector store, retriever, generator, ingestion API.

Net-new

Loading Orchestrator + type adapters, Telemetry Collector + distillation, Config Store, Config Resolver, Tuning Controller, Eviction worker, Tenant→DB catalog, Admin console.

The only net-new component with no precedent in a standard RAG is the Tuning Controller — deliberately the last thing built, so the system is useful (manually configurable) well before it's autonomous.
Database layer

Postgres + pgvector schema

Two roles: a single Control DB (schema control) owning all configuration, audit, proposals, and catalogs; and a Knowledge Store (schema kb) deployed in one global database and one database per tenant.

(A) Control DB — schema control

  • tunable_parameter — the registry: one row per tunable, definition + default + bounds.
  • parameter_value — effective values per scope; learning_enabled is the Locked/Learning check.
  • parameter_change_log — append-only evidence substrate for rollback.
  • tuning_proposal — pending proposals for apply_behavior='propose'.
  • tuning_run — controller execution audit.
  • tenant_catalog — tenant → database routing; secret referenced, not stored (BYOV).
  • embedding_model_version — advisory swap + versioning.

(B) Knowledge Store — schema kb

  • kb_chunk — the vector table (write-mostly-once): embedding vector(1536), knowledge_type, namespace, version chain via supersedes / superseded_by.
  • pattern_stats — hot-updating 1:1 side table for Type 2; generated failure_rate; read by the eviction worker.
Global DB holds sor_doc, pattern, semantics, global telemetry_insight. Tenant DB holds customization + tenant telemetry_insight — same kb_chunk shape, different physical database.

Resolver function — effective value with precedence

Resolution order: tenant override → global override → definition default. Precedence: user lock > learned/global value > default.

-- Precedence:  user lock > learned/global value > default.
CREATE OR REPLACE FUNCTION resolve_parameter(p_key text, p_tenant text DEFAULT NULL)
RETURNS TABLE (effective_value jsonb, source value_source,
               learning_enabled boolean, locked boolean)
LANGUAGE plpgsql STABLE AS $$
BEGIN
    -- tenant-scoped override
    IF p_tenant IS NOT NULL THEN
        RETURN QUERY SELECT pv.current_value, pv.source, pv.learning_enabled,
               (pv.learning_enabled = false)
        FROM parameter_value pv
        WHERE pv.param_key = p_key AND pv.scope_type = 'tenant'
               AND pv.tenant_id = p_tenant;
        IF FOUND THEN RETURN; END IF;
    END IF;
    -- global override, then definition default …
END;
$$;
-- the vector table (cosine ANN via HNSW; partial per-type indexes)
CREATE TABLE kb_chunk (
    id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    knowledge_type      knowledge_type NOT NULL,
    namespace           text NOT NULL,        -- e.g. 'kb:patterns:global'
    content             text NOT NULL,
    embedding           vector(1536) NOT NULL,  -- match embed_model_version dims
    embed_model_version text NOT NULL,
    supersedes          uuid REFERENCES kb_chunk(id),  -- version chain
    superseded_by       uuid REFERENCES kb_chunk(id),
    confidence          real,                  -- telemetry_insight decay
    metadata            jsonb NOT NULL DEFAULT '{}'
);
CREATE INDEX ix_chunk_ann ON kb_chunk USING hnsw (embedding vector_cosine_ops);

Full DDL — enums, indexes, pattern_stats, seed rows, and the offboarding notes — lives in anvizent-rag-addon-schema.sql.

Sequenced delivery

Build plan

Sequenced by value delivery and dependency, not calendar. Ship Milestone 1 first and alone — it delivers a manually tunable RAG and de-risks everything downstream. Each later milestone adds one capability without reworking the last.

M1

Manually tunable RAG

Every parameter is live-configurable by hand through the Admin console. Nothing self-tunes yet — the "advanced users can tweak" half works end to end.
  • Apply control-DB migration (the schema SQL)
  • Config Store API — CRUD + resolve over the SQL function
  • Config Resolver SDK — in-process cache, fail-open
  • Read-seam injection — swap constants behind one middleware
  • Admin console (minimal) — Locked/Learning check, set value, list
Gate: the parameter inventory — which constants in the retrieval path become tunable. A 1-hour decision that unblocks the milestone; get it wrong and the read-seam work churns.
M2

Signal flowing

Telemetry is collected and distilled into Type 5 insights landing in the store. Nothing acts on it yet, but the loop has fuel.
  • Telemetry seam — emit outcome on generate
  • Telemetry Collector + queue/topic
  • Distillation — rules gate → LLM phrasing → insight vectors
Gate: the rule-gate thresholds and the insight-statement schema — what counts as "signal." The intellectual core of Type 5; the code is trivial once defined.
M3

Self-tuning

The Locked/Learning check now does something. Auto-apply parameters move within bounds; propose parameters queue for approval.
  • Tuning Controller — objective compare, bounded delta, rate-limit
  • Shadow / A-B evaluation harness + auto-rollback
  • Proposal approval flow in Admin console
Gate: per-parameter objective functions and bounds. Run the controller in shadow mode and watch it before enabling auto-apply — that observation period is real calendar time and shouldn't be compressed.
M4

Loading discipline (write seam)

The five knowledge types flow through the orchestrator with correct metadata stamping, regimes, and tenant scoping.
  • Loading Orchestrator core — metadata stamp, ingestion wrap
  • Adapter: connector-metadata doc corpus (1a)
  • Adapter: live-schema cache, store-bypass (1b)
  • Adapter: pattern-capture hook on deploy-success (2)
  • Adapter: semantics batch-diff intake (3)
  • Adapter: customization capture-on-save (4)
Gate: adapters for patterns (2) and customizations (4) depend on hooks in the existing platform (deploy-success, artifact-save). Where those don't exist yet, that integration is the time sink — not the adapter code.
M5

Hygiene + tenancy hardening

The index can't rot, and database-per-tenant is operationally real.
  • Eviction / Hygiene worker — supersede/decay scans
  • pattern_stats update path from telemetry
  • Tenant→DB catalog wiring, provisioning, fan-out
Gate (infra): tenant-DB provisioning is the one genuinely multi-hour-to-day item — connection pooling across many DBs, migration fan-out, provisioning/offboarding automation. The only place the hours-not-days rule bends.

Parallelization

Two independent tracks once Milestone 1 lands. They share only the schema (M1.1) and the telemetry collector (M2.2 feeds M5.2) — run them in parallel worktrees; they don't block each other.

Track A (tuning): M2 → M3. Touches the read and telemetry seams.
Track B (loading/hygiene): M4 → M5. Touches the write seam and the store.

Steps M1–M2 alone deliver a manually tunable RAG; M3 makes it self-tuning; M4–M5 are loading discipline and hygiene. Calendar time is set by the design gates, the shadow-mode observation, the tenant-DB infra, and existing-platform hook availability — not by the coding.
